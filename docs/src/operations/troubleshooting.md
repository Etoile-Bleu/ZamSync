# Troubleshooting

Common issues and their resolution. For each entry, the symptom appears as you would see it in logs or terminal output.

---

## Sync: connection refused

**Symptom**

```
sync warn:  attempt 1/5 failed (connection refused), retrying in 100ms
sync warn:  attempt 5/5 failed (connection refused), retrying in 800ms
```

**Causes and fixes**

1. The hub's `zamsync serve` process is not running. Check with `systemctl status zamsync` on the hub and restart if needed.
2. Wrong port. Verify the bind address in the hub's environment file (`/etc/zamsync/zamsync.env`) and in the clinic's `zamsync sync` or `daemon` command.
3. Firewall blocking the connection. Check that port 7000 (or your configured port) is open from the clinic's IP to the hub:

```sh
# From the clinic:
nc -zv <hub-ip> 7000
```

4. Hub is behind a NAT or load balancer that has not been configured for TCP passthrough. ZamSync uses a raw TCP framing protocol -- HTTP proxies will not work.

## Sync: TLS handshake error

**Symptom**

```
sync warn:  attempt 1/5 failed (tls handshake error: ...), retrying in 100ms
```

**Causes and fixes**

1. **Clinic cert not signed by hub CA.** The clinic's `tls/node.crt` must be signed using `zamsync sign <clinic-dir> --ca <hub-dir>`. Re-run sign and redistribute the credentials.
2. **Hub started without `--tls`.** If the hub is running plain TCP and the clinic sends `--tls`, the handshake fails immediately. Either add `--tls` to the hub's `ExecStart`, or remove it from the clinic's sync command.
3. **Certificate expired.** TLS certificates generated by `keygen` and `sign` have a validity period. Check with:

```sh
openssl x509 -in /var/lib/zamsync/tls/node.crt -noout -dates
```

Re-run `keygen` on the hub and re-run `sign` for each clinic to issue fresh certificates. Distribute the new `ca.crt` to all clinic nodes.

4. **Node ID mismatch.** The `<peer-id>` argument to `zamsync sync` must match the hub's node identity. Read it from the hub:

```sh
zamsync info /var/lib/zamsync
# node_id  :  1234567890
```

Then use that value on the clinic:

```sh
zamsync sync /var/lib/clinic 192.168.1.10:7000 1234567890 --tls
```

## WAL_UNAVAILABLE on /submit

**Symptom**

```json
{"error": "WAL_UNAVAILABLE", "message": "IO error: No space left on device (os error 28)"}
```

**Causes and fixes**

| Cause | Fix |
|-------|-----|
| Disk full | Free space: run `zamsync expire` to drop old events, or move the data directory to a larger filesystem. |
| Data directory deleted or unmounted | Restore the directory and restart the service. |
| Filesystem mounted read-only | Check `dmesg` for disk errors. Remount: `mount -o remount,rw /`. |
| Permission denied | Ensure the `zamsync` user owns the data directory: `chown -R zamsync:zamsync /var/lib/zamsync` |

After resolving the underlying problem the server will recover automatically -- no restart is required. Clients should retry with exponential backoff (see [Error Codes](../error-codes.md)).

## No events visible after sync

**Symptom**

You ran `zamsync sync` successfully but `zamsync audit` on the hub shows no events from the clinic.

**Checklist**

1. **Wrong data directory.** Confirm you are auditing the hub's data directory, not a clinic's:

```sh
zamsync info /var/lib/zamsync
# Check that the node_id and event count are what you expect.
```

2. **Policy is `own`.** If the hub is running with `--policy own`, clinics only receive their own events back -- but the hub still stores all events from all clinics. Audit should show them:

```sh
zamsync audit /var/lib/zamsync --node <clinic-node-id>
```

3. **Sync direction.** `zamsync sync` is bidirectional, but the clinic must have had events to send. If the clinic WAL was empty, the hub receives nothing. Submit an event on the clinic first, then sync.

4. **Key file mismatch.** If the hub WAL is encrypted, `audit` requires `--key-file`:

```sh
zamsync audit /var/lib/zamsync --key-file /etc/zamsync/data.key
```

## Prometheus metrics not appearing

**Symptom**

`curl http://hub:9090/metrics` returns an empty response or connection refused immediately after startup.

**Causes and fixes**

1. **`--metrics` flag not set.** The Prometheus endpoint is disabled by default. Add `--metrics 0.0.0.0:9090` to the serve or daemon command.
2. **Counters are zero at startup.** Most metrics (`zamsync_sync_sessions_total`, `zamsync_events_sent_total`, etc.) start at zero and only appear in the output once they have been incremented. Trigger a sync session and then scrape again.
3. **Wrong bind address.** The metrics endpoint binds on whatever address you pass to `--metrics`. If you used `127.0.0.1:9090`, it will only be reachable from localhost. Use `0.0.0.0:9090` to expose it to the network.

## Budget exhausted before sync completes

**Symptom**

```
sync done:  sent=83  received=0  bytes=524288  budget exhausted -- resume with next sync
```

**Explanation**

This is expected behavior when `--max-bytes` is set below the total data to transfer. ZamSync sends events in batches and stops when the byte budget is reached. The remote peer's version vector is updated to reflect what was received, so the next sync resumes from where this one left off -- no events are lost or duplicated.

**If budget exhausts immediately (sent=0)**

The `--max-bytes` value is smaller than a single batch. Increase it or remove the cap:

```sh
# Remove the cap entirely for this sync
zamsync sync /var/lib/zamsync hub.example.com:7000 1234567890

# Or increase to 10 MB
zamsync sync /var/lib/zamsync hub.example.com:7000 1234567890 --max-bytes 10M
```

## Events appear out of order

**Symptom**

`zamsync audit` output shows events from multiple nodes interleaved in a seemingly random order.

**Explanation**

This is not a bug. `audit` uses `sorted_scan`, which merges per-node event streams by HLC timestamp in causal order. Events that were submitted on different nodes at approximately the same wall-clock time can appear in any order relative to each other -- the HLC timestamp and tie-breaking node ID determine the canonical order.

If you need events in strict per-node submission order, filter by node:

```sh
zamsync audit /var/lib/zamsync --node 2748582051
```

For state projection (building a read model from the full event log), always use `sorted_scan` order -- it guarantees that events a node knew about when it submitted its own events are earlier in the sequence.

## Version vector drift

**Symptom**

The `zamsync_vv_drift_events` metric stays high. `zamsync info` shows two nodes diverging.

**Explanation**

VV drift counts how many events are in the remote node's WAL that the local node has not yet received. A persistently high value means the two nodes have stopped syncing, or sync sessions are not completing.

**Checklist**

1. Check that the daemon or sync cron job is running on the clinic: `systemctl status zamsync-daemon`.
2. Look for sync errors in the journal: `journalctl -u zamsync-daemon --since "1 hour ago"`.
3. Verify network connectivity between the clinic and hub.
4. If budget is capped (`--max-bytes`), increase it or ensure sync runs frequently enough to drain the backlog.

## WAL recovery after power loss

**Symptom**

ZamSync logs a warning at startup like:

```
wal: truncated to last valid record at offset 81920
```

**Explanation**

This is normal and expected. The WAL is append-only, and ZamSync uses CRC32 checksums to detect partial writes. When the process is killed (power loss, OOM, crash) mid-write, the incomplete record at the tail of the file has a wrong or missing checksum. On the next open, the engine scans backward to the last valid record, truncates the file there, and continues normally.

No manual recovery step is required. Events that were in flight at the time of the crash are not visible, but events that completed (`engine.sync()` returned `Ok`) are preserved.

## Encrypted WAL cannot be audited

**Symptom**

```
Error: unexpected EOF or bad magic
```

when running `zamsync audit /var/lib/zamsync`.

**Fix**

Provide the key file:

```sh
zamsync audit /var/lib/zamsync --key-file /etc/zamsync/data.key
```

All WAL-reading commands (`audit`, `expire`, `snapshot`, `info`, `project`) require `--key-file` if the WAL was created with `keygen` encryption. Only `compact` does not support `--key-file`.
