# CLI Reference

ZamSync ships a single binary, `zamsync`, with subcommands for every operation.
All commands write to stdout; errors and warnings go to stderr.

## Global flags

These flags are accepted by every command that opens the WAL.

| Flag | Description |
|------|-------------|
| `--key-file <path>` | Decrypt the WAL with the key at `<path>` (generated by `keygen`). Required if the WAL was encrypted at creation time. |
| `--schema <mode>` | Payload validation mode: `none` (default), `json`, or `json+field1,field2` (JSON with required fields). |
| `--no-color` | Disable ANSI color output. Also honored via `NO_COLOR=1` or `TERM=dumb`. |

---

## info

Print a summary of a local data directory.

```
zamsync info <data-dir> [--key-file <path>]
```

**Output**

```
node_id  :  <id>
data_dir :  <path>
events   :  <count>
vv       :  node <id> @ seq <seq>
wal size :  <N> KB
oldest   :  <YYYY-MM-DD>
newest   :  <YYYY-MM-DD>
         :  peers:
    node <id>    <N> events
```

The version vector (`vv`) has one line per tracked peer. If the WAL is empty, `oldest` and `newest` show `--` and `peers` shows `(none)`.

---

## submit

Write a single event to the local WAL.

```
zamsync submit <data-dir> <payload> [--key-file <path>] [--schema <mode>]
```

`<payload>` is a raw byte string. The event type is always `1`.

**Output**

```
submitted seq=<N>
```

---

## sync

Perform a one-shot bidirectional sync with a remote peer.

```
zamsync sync <data-dir> <peer-addr> <peer-id> [--tls] [--key-file <path>] [--schema <mode>] [--max-bytes <N>] [--metrics <addr>]
```

| Flag | Description |
|------|-------------|
| `--tls` | Use mTLS with credentials in `<data-dir>/tls/`. |
| `--max-bytes <N>` | Cap the number of bytes sent in this session. Accepts plain integers or a `K`, `M`, `G` suffix (e.g. `2M`, `512K`). Useful on metered links. |
| `--metrics <addr>` | Expose a Prometheus `/metrics` endpoint on `<addr>` (e.g. `0.0.0.0:9090`). |

**Output -- success**

```
sync done:  sent=<N>  received=<N>  bytes=<N>
```

If `--max-bytes` was reached before the session completed:

```
sync done:  sent=<N>  received=<N>  bytes=<N>  budget exhausted -- resume with next sync
```

**Output -- transient failure (retrying)**

```
sync warn:  attempt <N>/5 failed (<error>), retrying in <ms>ms
```

The command retries up to 5 times with exponential backoff starting at 100 ms (100 ms, 200 ms, 400 ms, 800 ms). A permanent error exits immediately.

## serve

Listen for incoming sync connections and handle peers concurrently.

```
zamsync serve <data-dir> <bind-addr> [--tls] [--key-file <path>] [--schema <mode>] [--policy all|own] [--max-peers <N>] [--retain <Nd>] [--metrics <addr>] [--http <addr>]
```

| Flag | Description |
|------|-------------|
| `--tls` | Use mTLS with credentials in `<data-dir>/tls/`. |
| `--policy all` | Send all events to any peer (default). |
| `--policy own` | Send only a peer's own events back to it. |
| `--max-peers <N>` | Maximum concurrent peer sessions (default: `16`). |
| `--retain <Nd>` | Drop events older than N days on startup (e.g. `365d`). |
| `--metrics <addr>` | Expose a Prometheus `/metrics` endpoint. |
| `--http <addr>` | Start the REST API server on `<addr>` (e.g. `127.0.0.1:8080`). |

**Output -- startup (plain TCP)**

```
node <id> listening on <addr> [policy=<policy>] [max-peers=<N>]
```

**Output -- startup (TLS)**

```
node <id> TLS listening on <addr> [policy=<policy>] [max-peers=<N>]
```

**Output -- per peer session (success)**

```
peer <id> done: sent=<N> received=<N>
TLS peer <id> done: sent=<N> received=<N>
```

The `TLS ` prefix appears when the connection used mTLS (`--tls`).


**Output -- retention on startup (when `--retain` drops events)**

```
retain   :  dropped <N> events, freed <N> KB
```

Errors are written to stderr: `peer <id> sync error: <message>`.

## daemon

Run a persistent sync loop against a single peer.

```
zamsync daemon <data-dir> <peer-addr> <peer-id> [--tls] [--key-file <path>] [--schema <mode>] [--interval <secs>] [--metrics <addr>]
```

| Flag | Description |
|------|-------------|
| `--tls` | Use mTLS with credentials in `<data-dir>/tls/`. |
| `--interval <secs>` | Seconds between sync attempts (default: `60`). |
| `--metrics <addr>` | Expose a Prometheus `/metrics` endpoint. |

**Output -- startup**

```
daemon: node <id> -- peer <id> (TLS|TCP) -- interval <N>s
```

**Output -- per sync cycle**

```
[sync] peer=<id>  sent=<N>  received=<N>
[sync] peer=<id>  already in sync
```

Errors are written to stderr: `[sync] peer=<id> error: <message>`.

The daemon runs indefinitely. Send `SIGINT` (Ctrl-C) to stop it.

---

## ping

Measure round-trip latency to a remote node.

```
zamsync ping <data-dir> <peer-addr> [--tls] [--count <N>] [--timeout <ms>]
```

| Flag | Default | Description |
|------|---------|-------------|
| `--tls` | -- | Use mTLS. |
| `--count <N>` | `3` | Number of probes to send. |
| `--timeout <ms>` | `5000` | Per-probe timeout in milliseconds. |

Each probe opens a fresh TCP connection, exchanges a Handshake message, and measures the total round-trip time including connection setup.

**Output**

```
PING <peer-addr>  local-node=<id>  [TLS]

  seq=1  peer=<id>  rtt=<N>ms  [tls=ok]
  seq=2  peer=<id>  rtt=<N>ms
  seq=3  error: <message>
---
<successes>/<total>  loss=<pct>%
rtt  min=<N>ms  avg=<N>ms  max=<N>ms
```

RTT values are color-coded: green below 100 ms, yellow below 500 ms, red at 500 ms and above. Loss rate is color-coded: green at 0 %, yellow at 50 % or below, red above 50 %.

Exits with code `1` if all probes fail.

## compact

Trim WAL records that all known peers have already received.

```
zamsync compact <data-dir>
```

There is no `--key-file` support; the WAL must be unencrypted to compact.

**Output -- nothing to compact**

```
nothing to compact (no peers have confirmed events yet)
```

**Output -- records dropped**

```
compacted: dropped <N> WAL records
```

---

## audit

Print a log of all events in the WAL for inspection and compliance.

```
zamsync audit <data-dir> [--format json|text|csv] [--since <unix-ms>] [--node <id>] [--key-file <path>]
```

| Flag | Description |
|------|-------------|
| `--format text` | Human-readable table with aligned columns (default). |
| `--format json` | One JSON object per line (NDJSON). |
| `--format csv` | CSV with header row. |
| `--since <unix-ms>` | Show only events with an HLC timestamp at or after `<unix-ms>`. |
| `--node <id>` | Show only events originating from node `<id>`. |

**Text output**

```
timestamp                    node        seq   type     size  sha256
------------------------------------------------------------------------------------------
2024-01-15T10:30:00.000Z       42         1      1      128  a3f1c9e2b4d7...
```

The SHA-256 column shows the first 16 hex characters. The full 64-character hash appears in JSON and CSV output.

**JSON output (one object per line)**

```json
{"ts":"2024-01-15T10:30:00.000Z","ts_ms":1705314600000,"node":42,"seq":1,"type":1,"size":128,"sha256":"a3f1c9e2...","hlc_logical":0}
```

## expire

Delete events older than a cutoff date.

```
zamsync expire <data-dir> --before YYYY-MM-DD [--dry-run] [--min-keep <N>] [--key-file <path>]
```

| Flag | Description |
|------|-------------|
| `--before YYYY-MM-DD` | Required. Expire all events with an HLC timestamp before this UTC date. |
| `--dry-run` | Preview how many events would be dropped without modifying the WAL. |
| `--min-keep <N>` | Keep at least the N most recent events regardless of age (default: `0`). |

**Output -- dry run**

```
dry-run  :  <N> events would be expired
payload  :  <N> KB in expirable payloads
wal size :  <N> KB
```

**Output -- nothing to drop**

```
expire   :  nothing to drop (all events newer than cutoff)
```

**Output -- events dropped**

```
expire   :  dropped <N> events, freed <N> KB
```

---

## snapshot

Copy the WAL to a file for backup or offline transfer.

```
zamsync snapshot <data-dir> --output <path> [--key-file <path>]
```

The WAL is flushed (`fsync`) before the copy to ensure all in-memory data is on disk.

**Output**

```
snapshot : <N> KB written to <path>
```

---

## rekey

Re-encrypt the WAL with a new key.

```
zamsync rekey <data-dir> --old-key <path> --new-key <path>
```

Both flags are required. The command reads the entire WAL with the old key, writes a temporary file `events.wal.rekey` in the same directory, then atomically replaces the original.

**Output**

```
Re-keyed <N> WAL records in <path>
Update your --key-file to point to the new key.
```

---

## keygen

Generate TLS credentials and a WAL encryption key for a hub node.

```
zamsync keygen <data-dir>
```

Creates `<data-dir>/tls/` and writes:

| File | Description |
|------|-------------|
| `ca.crt` | CA certificate -- copy to all nodes in the deployment. |
| `ca.key` | CA private key -- keep secret; used only to sign new node certs. |
| `node.crt` | This node's identity certificate. |
| `node.key` | This node's private key -- never share. |
| `data.key` | 32-byte random key for WAL encryption (ChaCha20-Poly1305). |

> **Security note:** Move `data.key` outside the data directory in production (e.g. `/etc/zamsync/data.key`, `chmod 600`). If the data directory is compromised, the key must not be stored inside it.

---

## sign

Sign a new node certificate for a clinic (or any non-hub node) using the hub CA.

```
zamsync sign <clinic-dir> --ca <hub-dir>
```

Alternatively, provide the CA files individually:

```
zamsync sign <clinic-dir> --ca-cert <path> --ca-key <path>
```

Creates `<clinic-dir>/tls/` and writes:

| File | Description |
|------|-------------|
| `ca.crt` | Hub CA certificate (same as the hub's `ca.crt`). |
| `node.crt` | Clinic node certificate, signed by the hub CA. |
| `node.key` | Clinic node private key -- never share. |
| `data.key` | A new WAL encryption key for this clinic node. |

**Output** includes ready-to-run `serve` and `sync` commands for the clinic device.

## setup

One-shot hub initialization: creates the data directory, generates TLS credentials and a WAL key, and writes a ready-to-install systemd unit file. It is a convenience wrapper around `keygen`.

```
zamsync setup --hub <data-dir> [--bind <addr>]
```

| Flag | Default | Description |
|------|---------|-------------|
| `--hub <data-dir>` | required | Path to the hub data directory (created if absent). |
| `--bind <addr>` | `0.0.0.0:5000` | Bind address written into the generated systemd unit. |

`setup` will refuse to run if `<data-dir>/tls/node.key` already exists to prevent accidental overwrite of existing credentials.

**Output**

```
Setting up hub in directory: /var/lib/hub
[ok] directory ready
[ok] keys generated
[ok] systemd unit created -> /var/lib/hub/zamsync-hub.service

Next steps:
  1. Copy the unit:
     sudo cp /var/lib/hub/zamsync-hub.service /etc/systemd/system/
  2. Enable + start:
     sudo systemctl enable --now zamsync-hub
  3. For each clinic node:
     copy /var/lib/hub/tls/ca.crt
     zamsync keygen ./clinic-data
     zamsync sign ./clinic_data --ca /var/lib/hub
  Bind address: 0.0.0.0:5000
```

The generated unit file starts `zamsync serve <data-dir> <bind-addr>` without TLS or encryption flags. To add mTLS or WAL encryption, edit the unit's `ExecStart` to include `--tls` and `--key-file` after moving `data.key` to a secure path -- see [Deployment](operations/deployment.md).

## bench

Run a local write-throughput and sync-bandwidth benchmark.

```
zamsync bench <data-dir> [--events <N>]
```

| Flag | Default | Description |
|------|---------|-------------|
| `--events <N>` | `10000` | Number of events to write. Each event has a 62-byte payload. |

The benchmark measures:

- **Submit throughput** -- events/sec for writing N events to the WAL.
- **WAL reload time** -- time to replay the WAL from disk (simulates startup).
- **Memory usage** -- resident set size after reload (target: under 100 MB).
- **Sync bandwidth (uncapped)** -- full sync between two in-process nodes over loopback TCP: events/sec and MB/s.
- **Sync bandwidth (capped at 10 %)** -- same sync with a byte budget set to 10 % of the uncapped wire size, to simulate a satellite or metered session.

**Output**

```
bench: <N> events, payload 62 bytes
data : <path>

=== submit ===
  time       : 0.123s
  throughput : 81300 events/sec
  wal size   : 512 KB

=== reload (WAL replay) ===
  time       : 0.045s

=== memory (after reload) ===
  rss        : 8192 KB (8 MB)  [target: <100 MB] -- OK

=== sync bandwidth ===
  [full sync -- no cap]
    events received : <N>
    wire bytes sent : 0.62 MB (<N> B)
    compression     : 1.0x (vs raw payload)
    time            : 0.211s
    throughput      : 47393 events/sec
    bandwidth       : 2.94 MB/s

  [capped sync -- budget <N> B (~10% of full)]
    events received : <N>
    wire bytes sent : <N> B
    budget exhausted: true
    time            : 0.021s
    progress        : 10.0% events delivered in this window
```

Memory reporting is supported on Linux and Windows. On other platforms it prints `(not available on this platform)`.

## PKI workflow

Complete example for a hub-and-clinic deployment with mTLS and WAL encryption.

**1. Initialize the hub**

```
zamsync keygen /data/hub
# Move the WAL key out of the data directory:
mv /data/hub/tls/data.key /etc/zamsync/hub.key
chmod 600 /etc/zamsync/hub.key
```

**2. Sign each clinic node**

```
zamsync sign /data/clinic1 --ca /data/hub
```

**3. Start the hub server**

```
zamsync serve /data/hub 0.0.0.0:7700 --tls --key-file /etc/zamsync/hub.key
```

**4. Start sync from a clinic**

```
zamsync sync /data/clinic1 192.168.1.10:7700 1 --tls \
  --key-file /data/clinic1/tls/data.key
```

**5. Rotate a WAL key**

```
zamsync rekey /data/hub --old-key /etc/zamsync/hub.key --new-key /etc/zamsync/hub-new.key
```

## Color output

All commands emit ANSI color codes when connected to a terminal. To disable:

| Method | Description |
|--------|-------------|
| `--no-color` | CLI flag, accepted by every command. |
| `NO_COLOR=1` | Environment variable (see [no-color.org](https://no-color.org)). |
| `TERM=dumb` | Environment variable (legacy terminal compatibility). |
