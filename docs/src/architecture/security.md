# Security Model

ZamSync has two independent security layers: **transport encryption** (mTLS) and **at-rest encryption** (ChaCha20-Poly1305). Each can be used independently. A deployment with an isolated private network may use only at-rest encryption; a deployment with untrusted network paths may use both.

---

## Threat model

| Threat | Mitigation |
|--------|-----------|
| Passive eavesdropping on the network | mTLS encrypts all sync traffic |
| Rogue node injecting events | mTLS mutual authentication: only nodes with a cert signed by the hub CA are accepted |
| Stolen or accessed disk image | WAL encryption: events are unreadable without the key file |
| Tampered WAL record on disk | ChaCha20-Poly1305 authentication tag + CRC32 detect modification |
| Partial write / disk corruption | CRC32 per record; WAL recovery truncates at last clean record |
| Unauthorized peer reading another clinic's events | `--policy own` on the hub limits each peer to its own events |

### What is not protected

- **Event metadata in the WAL header is unencrypted.** The sequence number, record length, and version byte are always plaintext. Only the payload (the serialized `Event` struct, including the application data) is encrypted.
- **`peers.state` is unencrypted.** The version vectors stored in `peers.state` reveal which nodes have synced and how many events each has submitted, even if the WAL is encrypted.
- **Traffic analysis.** mTLS hides the content of sync sessions but not their timing, duration, or byte volume to a passive observer watching the network.
- **Key-in-memory attacks.** The decryption key is loaded into process memory during operation. An attacker with access to the process's memory space can extract the key.

## Transport security: mTLS

ZamSync uses mutual TLS (mTLS) for all network communication when `--tls` is passed. Both the client (initiator) and the server (responder) present a certificate. The connection is rejected if either certificate is not signed by the same CA.

### Certificate hierarchy

```
Hub CA (ca.crt / ca.key)
    │
    ├── Hub node cert (node.crt / node.key)   -- generated by keygen
    │
    ├── Clinic 1 cert (node.crt / node.key)   -- generated by sign
    ├── Clinic 2 cert (node.crt / node.key)   -- generated by sign
    └── ...
```

The hub CA is self-signed. All node certificates are signed by the hub CA. There is no intermediate CA.

### Key generation

`zamsync keygen <hub-dir>` generates all hub credentials in one step:

- `ca.crt` / `ca.key`: the hub CA, used to sign new node certificates
- `node.crt` / `node.key`: the hub node's own identity certificate
- `data.key`: 32-byte random WAL encryption key

`zamsync sign <clinic-dir> --ca <hub-dir>` generates clinic credentials:

- Copies `ca.crt` from the hub (so the clinic trusts the same CA)
- Generates a new `node.crt` / `node.key` signed by the hub CA
- Generates a new `data.key` for this clinic's WAL

Each clinic gets its own WAL key. Compromising one clinic's key does not expose any other clinic's WAL.

### Runtime

The TLS implementation uses [rustls](https://github.com/rustls/rustls) with a ring crypto provider. Cipher suite and protocol version negotiation are handled by rustls; ZamSync does not configure them explicitly beyond providing the certificate and key.

## At-rest encryption: ChaCha20-Poly1305

WAL records are encrypted individually using ChaCha20-Poly1305, an Authenticated Encryption with Associated Data (AEAD) cipher. ChaCha20 is the stream cipher; Poly1305 is the authentication tag.

### Key

The key is a 32-byte (256-bit) random value generated by `keygen` and stored as a raw binary file (`data.key`). The key file has no header or framing; it is exactly 32 bytes.

Load the key with `--key-file <path>`. If the WAL was created with a key and the key is not provided, the engine returns a configuration error rather than treating encrypted records as corruption.

### Per-record nonce

A fresh 96-bit (12-byte) nonce is generated from the OS random source for every record. Reusing a nonce with the same key under ChaCha20-Poly1305 would break confidentiality, so ZamSync never reuses nonces even across process restarts.

### Encrypted payload format

```
[ nonce: 12 bytes ][ ciphertext: N bytes ][ tag: 16 bytes ]
```

The ciphertext is the rkyv-serialized `Event` encrypted with ChaCha20. The 16-byte Poly1305 tag authenticates the ciphertext and nonce together. Decryption fails with an explicit error if the tag does not match, covering both tampering and wrong-key scenarios.

### CRC32 and the authentication tag

CRC32 and the authentication tag protect against different failure modes:

| Check | Detects |
|-------|---------|
| CRC32 (in WAL header) | Accidental bit flips, filesystem corruption, partial overwrites |
| Poly1305 tag (in encrypted payload) | Intentional tampering, wrong decryption key |

The CRC32 covers the encrypted bytes, not the plaintext. An attacker who modifies ciphertext bytes will be caught by the tag; an attacker who modifies the CRC32 field will be caught by the mismatch between the stored CRC and the computed CRC.

### Key rotation

`zamsync rekey <data-dir> --old-key <path> --new-key <path>` re-encrypts the entire WAL with a new key. The command:

1. Reads every record using the old key.
2. Writes a new file `events.wal.rekey` in the same directory, encrypting each payload with the new key and fresh nonces.
3. Atomically replaces `events.wal` with the re-keyed file.

The `peers.state` file is not re-encrypted (it is always plaintext). Update `--key-file` to point to the new key before restarting any process that opens the WAL.

## Access control: policy

The `--policy` flag on `zamsync serve` controls which events the hub sends to connecting peers:

**`--policy all` (default).** Every peer receives all events. Use this for a single-tenant deployment or when all clinic nodes are trusted with the full dataset.

**`--policy own`.** Each peer receives only the events originally submitted by that peer's node ID. Use this for multi-tenant hubs where each clinic should not see other clinics' data. This is a server-side filter: the hub holds all events but returns only the requesting peer's subset.

Note that `--policy own` does not prevent a client from requesting arbitrary events over a raw TCP connection. mTLS must also be enabled so that the hub can authenticate the peer's node identity before applying the policy filter.

## Production hardening checklist

- [ ] Move `data.key` outside the data directory: `/etc/zamsync/data.key`, `chmod 600`, owned by the service account.
- [ ] Never commit `data.key`, `ca.key`, or `node.key` to version control.
- [ ] Use `--policy own` on multi-tenant hubs and enable `--tls`.
- [ ] Rotate WAL keys periodically with `zamsync rekey`.
- [ ] Restrict the data directory permissions so only the service account can read `events.wal` and `peers.state`.
- [ ] Back up `data.key` separately from the WAL. A lost key makes the encrypted WAL permanently unreadable.
